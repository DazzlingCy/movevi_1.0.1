# MOVEVI 木卫六 App - V1.1 迭代执行计划

基于“让跑步机从一次性硬件消费，变成持续内容体验”的战略方向，本轮 V1.1 版本（代号：**“激活与首跑”**）旨在把 App 从宏大的“世界探索观”向拉动硬件复跑率的“内容消费引擎”转型。

## 🎯 核心目标 (North Star Metric)

**每台已激活跑步机的月均实景跑步时长**

放弃泛社区拉新，不看 App 的独立下载量，聚焦设备持有者的使用频次：
- **目标1：** 提升设备一次性激活率与设备绑定覆盖率。
- **目标2：** 提升首次 10-15 分钟路线的起跑与完赛转化。
- **目标3：** 以明确的“跑后结算”推动复跑产生（Next-Run引导）。

---

## 🛠️ V1.1 核心重构模块与技术方案

### 1. 强制激活与设备绑定链路 (Device Binding Onboarding)
*当前状态：跑步机连接散落在首页左上角非强制区域。*

**迭代计划：**
- **开机与首测屏强引导：** 改造 `IntroScreen` 或应用首次加载逻辑，加入全屏的蓝牙扫描与设备绑定流程。
- **沉没成本与专属价值构建：** 设备一旦绑定，在 UI 层弹出专属通知“您的专属实景设备已连接，当前光迹值翻倍”。
- **硬件强关联：** 确保非绑定状态的用户体验降级，引导必须在实体设备上发生核心行为，将个人主页从普通的社会化页面改为**资产与设备中心 (ProfileTab改版)**。

### 2. 首页：“今天跑什么”替代“世界地图” (Action-Oriented Dashboard)
*当前状态：首页是一个可拖拽的世界地图，过于宏大导致单次决策成本高。*

**迭代计划：**
首页重构，将地图置于底层背景（保留视效），主视角提至前端的“动作面板层”（Action Cards）：
- **[最高优] 继续上次路线（Continue Last Run）：** 提取 `userStats` 中的 `lastRun` 记录，一键恢复上次未跑完的城市路网，不经过二次挑选。
- **[纯新手] 15 分钟轻松跑（Quick 15 Min Run）：** 特设低门槛入口，绕过城市路线选择，直接一键开始系统设定好的友好路段（如河畔、平原）。
- **[运营位] 本周热门与点亮目标：** 展示当前正被全网集中点亮的城市（如“正在点亮：东京”）。将长期的“点亮地球计划”具象为明确的“本周任务”。

### 3. 正反馈与闭环结算池 (Post-Run Settlement Loop)
*当前状态：跑后结算可能仅显示通用数据（距离、卡路里），对下一次的刺激不足。*

**迭代计划：**
- **仪式感结算屏：** 每次实景跑结束时（`RunPlaybackView` 结束时刻），强调该城市的进度推进（如“城市碎片收集 +1”、“该城探索度到达 30%”）。
- **直接的下一次推荐：** 在结算页面直接展示推荐的“下一段路线卡片”，将用户的完成感立刻转化为对下一次的期待，形成**复跑闭环**。
- **光力值机制明确化：** 跑步生成的卡路里或距离，统一转化为“光力值”，该数值用于后续解锁被锁定的世界城市。

## 📅 排期与实施节奏 (Execution Timeline)

| 阶段 | 里程碑 | 开发任务 | 验证指标 |
| :--- | :--- | :--- | :--- |
| **第一阶段** | 首页架构翻新 | 重写 `HomeTab.tsx`，分离地图底层与卡片操作层，加入“继续前行”、“15分钟跑”等快捷入口 | 首页到播放器的转化时间缩短<br/>首次开跑率提升 |
| **第二阶段** | 结算闭环与推荐 | 改造 `RunPlaybackView.tsx`，加入城市片断解锁动效并渲染“下一次推荐”卡片 | 跑后 7日内重跑率提升 |
| **第三阶段** | 绑定与资产化 | 强化 `ProfileTab` 及全局顶部的设备绑定流程与 UI | 设备实名/账户绑定率达 85%+ |

> **开发说明：** 当前前端技术栈为 React (Vite) + Tailwind CSS + motion 动画库。本迭代应充分利用 `motion` 实现卡片的丝滑视效，并将部分繁琐的地图拉拽操作降维为固定视角+内容消费卡片。
